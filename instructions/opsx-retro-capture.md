PR/CI/レビューの証跡を収集し、事実ベースの振り返り入力を作成せよ。

**やること:**

1. 入力から対象を特定する
   - PR URL/番号
   - 失敗したCI run URL/ID
   - `takt run id`（必須）
   - レビューコメント（あれば）
   - 比較対象（期待仕様、OpenSpec成果物）
   - `takt run id` が未指定の場合:
     - `.takt/runs/` から候補runを列挙する
     - PR日時・ブランチ名・更新時刻で最有力候補を提示し、`証跡不足で分析不能` として終了する

2. 事実を収集する
   - PRの目的、変更ファイル、差分規模
   - 失敗ジョブと失敗ステップ（lint/test/build/review）
   - 失敗ログの主要エラー
   - 既存の`takt`レポート（`00-plan.md`, `coder-scope.md`, `supervisor-validation.md` 等）
   - `.takt/runs/{run_id}/logs/*.jsonl` から遷移監査ログを抽出する:
     - `piece_start` / `piece_complete`
     - `step_start`
     - 判定イベント（`REJECT` / `APPROVE` / `COMPLETE`）
     - ルール判定に使われたcondition文言
   - 主要抜粋を時系列で `retro-capture.md` に記録する

3. 期待との差分を抽出する
   - 「何を達成すべきだったか」
   - 「実際に出た結果」
   - その差分を観点ごとに列挙

4. 証跡の信頼度を判定する
   - 直接証跡（ログ/ファイル/レポート）
   - 推測（根拠が弱いもの）

5. `retro-capture.md` を出力する

**ルール:**
- 推測と事実を混ぜない
- ログ行・ファイルパス・レポート名で根拠を明示する
- 原因分析は次ステップで行い、このステップでは断定しない
- `.takt`ログの遷移証跡がない場合は「監査不能」として明示する

**完了条件:**
- 失敗事象を再現可能な単位で説明できる証跡が揃っている
- `.takt`ログに基づく遷移監査（どのmovementでどう判定されたか）が記録されている
