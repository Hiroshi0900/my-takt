tasks.md を読み込み、未完了タスクを分析してバッチ実行計画を立てよ。

**注意:** Previous Responseがある場合は前回バッチの完了後の再計画である。
tasks.md のチェック状況を確認し、次のバッチを決定せよ。

**やること:**
1. 対象feature名を特定する
   - 初回（00-plan.md がない）: User Task から feature 名を抽出する
   - 再計画（00-plan.md がある）: 既存 00-plan.md の「## 対象feature」を優先する
   - 特定できない場合は「要件が不明確、情報不足」と判定する
2. `tasks_path = openspec/specs/{feature}/tasks.md` を確定する
3. `{tasks_path}` を読み込む
   - ファイルが存在しない場合は「tasks.mdが存在しない、または空」と判定する
   - ファイルが空、またはタスク（`- [ ]` / `- [x]`）が1つもない場合も同様
4. 未完了タスク（`- [ ]`）を特定する
5. `PRバッチ計画` セクション（`## PRバッチ計画`）があるか確認する
   - ある場合は、各 `PR-*` の対象タスクと完了条件を抽出する
   - 既に完了済みのPRバッチがあればスキップし、最初の未完了PRバッチを「現在のPRバッチ候補」とする
6. 全タスクが完了済み（`- [x]`）の場合、「全タスク完了」と判定する
7. 未完了タスクがある場合:
   a. 現在のPRバッチ候補があり、その対象タスクが全て完了している場合:
      - 「PRバッチ完了（残タスクあり、今回セッションを完了してよい）」と判定してよい
      - その場合、今回のバッチは空にせず、どのPRバッチが完了したかをレポートに明記する
   b. 現在のPRバッチ候補が未完了の場合は、そのPRバッチ内のタスクから今回バッチを選ぶ
   c. `PRバッチ計画` がない場合は、従来どおり未完了タスク全体から今回バッチを選ぶ（後方互換）
   d. 各タスクの実装スコープを見積もる（変更ファイル数・複雑さ）
   e. コードを調査し、影響範囲を特定する
   f. バッチを決定する:
      - 同じファイルを触るタスクは同一バッチに集約
      - 1バッチは関連ソースコードがコンテキストに収まる範囲に制限
      - 原則として別PRバッチのタスクを混在させない
8. 実装アプローチを決める
   - ナレッジ・ポリシーの制約に違反しないか照合する

**PRバッチ優先ルール（追加）:**

- `PRバッチ計画` がある場合、まず未完了の最初のPRバッチを対象とする
- そのPRバッチが大きい場合は、同一PRバッチ内の部分集合を「今回のバッチ」としてよい
- 今回セッションの目的が「小さめPR完了」であり、現在のPRバッチ完了条件を満たした場合は `全タスク完了` でなくても終了可能

**バッチサイズの判定基準（両方の条件を満たすこと）:**

条件1: コンテキスト上限（関連コードがコンテキストに収まる最大タスク数）
- Small（1-2ファイル変更）: 最大8タスク
- Medium（3-5ファイル変更）: 最大3タスク
- Large（6+ファイル変更）: 最大1タスク

条件2: 最低作業量（起動オーバーヘッドを償却するための下限）
- 1バッチの推定作業時間が5分以上になるようタスクをまとめる
- Smallタスクのみの場合は最低3タスクを1バッチに含める
- 未完了タスクが下限に満たない場合は、残り全てを1バッチにまとめる

**必須出力:**

```markdown
# バッチ実行計画

## 対象feature
- feature: {feature}
- tasks_path: `openspec/specs/{feature}/tasks.md`

## 進捗
- 完了: {完了タスク数}/{全タスク数}
- 未着手: {未着手タスク数}

## PRバッチ進捗
- 現在のPRバッチ: {PR-1 / PR-2 / なし}
- PRバッチ完了条件: {要約}
- PRバッチの状態: 未着手 / 進行中 / 完了 / 該当なし

## 今回のバッチ
| タスクID | タスク概要 |
|----------|-----------|
| X.X | ... |

## 実装アプローチ
{バッチ内タスクの実装方針}

## 実装ガイドライン
- {Coderが従うべきガイドライン}
- 並行可能なタスクがある場合、CLIエージェントの内部並行処理機能を活用してよい
```
