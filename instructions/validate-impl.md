実装が要件・設計・タスクに合致しているかを検証せよ。

**注意:** `openspec/specs/{feature}/` に要件・設計・タスクが揃っていない場合はABORTする。

**やること:**
1. タスクから対象feature名を特定する
2. `openspec/specs/{feature}/` の全成果物を読み込む:
   - `requirements.md` - 要件
   - `design.md` - 設計
   - `tasks.md` - タスクリスト
3. タスク完了状況を確認する:
   - `tasks.md` のチェックボックス `[x]` を数える
   - 未完了タスクを特定する
4. 要件カバレッジを検証する:
   - 各要件IDに対応する実装コードが存在するか確認する
   - 受け入れ条件に対応するテストが存在するか確認する
   - 各要件IDに対して、実装箇所（ファイル:行）とテストケース（ファイル:テスト名）を1:1で紐付ける
   - 紐付け漏れがある場合は **検証不合格**
5. 設計整合性を検証する:
   - 設計のコンポーネント構成と実装が一致するか確認する
   - インターフェース定義が守られているか確認する
6. 品質ゲートコマンドを特定する（リポジトリ定義を優先）:
   - 優先順: `make` / `task` / `just` / package script / 直接コマンド
   - `lint` / `test` は必須。どちらかを特定できない場合は **検証不合格**
   - `build` は該当する場合に確認する
7. lint実行（必須）:
   - `lint` コマンドを実行し、成功を確認する
   - 失敗した場合は **検証不合格**
8. テスト実行（必須）:
   - テストコマンドを実行し、全テストがパスすることを確認する
   - 失敗した場合は **検証不合格**
9. ビルド実行（該当する場合）:
   - ビルドコマンドを実行し、成功を確認する
   - 失敗した場合は **検証不合格**
10. リグレッション確認:
   - 既存テストが影響を受けていないか確認する
11. OpenAPI契約整合チェック（該当APIがある場合は必須）:
   - 成功系: ステータスコードとレスポンス型の整合を確認する
   - 失敗系: ステータスコードとエラー応答型の整合を確認する
   - 差分がある場合は **検証不合格**
12. 重要度別の未解消指摘を集計する:
   - Critical / High / Warning を明示し、未解消件数を算出する
   - Critical / High が1件でも未解消なら **検証不合格**
   - Warning は許容理由がない限り **検証不合格**
13. 再レビュー（前回指摘の解消確認）:
   - 前回指摘ごとに「解消/未解消/許容」を判定する
   - 未解消がある場合は **検証不合格**

**判定ルール（追加）**
- `lint` と `test` のいずれかが未特定・未実行・失敗の場合、`検証合格` を選んではならない
- `lint` / `test` コマンドの探索内容（Makefile / package scripts / CI 等）はレポートに記録する
- プロジェクト標準の build コマンド（例: `go build ./...` / `npm run build`）と対象テストの実行証跡（実行コマンド、結果、失敗時の対応方針）がない場合、`検証合格` を選んではならない
- 要件トレーサビリティ（実装箇所 + テスト）の紐付け漏れがある場合、`検証合格` を選んではならない
- OpenAPI差分が残る場合、`検証合格` を選んではならない
- Critical / High の未解消指摘がある場合、`検証合格` を選んではならない
- Warning に許容理由がない場合、`検証合格` を選んではならない
