**これは {movement_iteration} 回目のAI Reviewです。**

初回は網羅的にレビューし、指摘すべき問題をすべて出し切ってください。
2回目以降は、前回REJECTした項目が修正されたかの確認を優先してください。

AI特有の問題についてコードをレビューしてください:

## レビュー観点

### 仮定の検証

```
[ ] 実装は実際に要求されたものと一致しているか（要件確認）
[ ] 既存コードベースの規則に合っているか（コンテキスト確認）
[ ] ビジネスルールが正しく理解されているか（ドメイン確認）
[ ] 現実的なエッジケースが考慮されているか
```

### もっともらしいが間違っているパターンの検出

```
[ ] 幻覚 API — 使用しているライブラリバージョンに存在しないメソッドの呼び出し
[ ] 古いパターン — 学習データからの非推奨アプローチの使用
[ ] 過剰エンジニアリング — タスクに不要な抽象化レイヤーの追加
[ ] 配線忘れ — 機構は実装されているが、エントリポイントから渡されていない
```

### Go 固有 AI アンチパターン

```
[ ] ファントム依存の検出:
    - go.mod / go.sum に存在しないパッケージのインポート
    - 存在しない標準ライブラリの関数呼び出し（例: strings.ContainsAny の誤用）

[ ] 不正な error wrapping:
    - fmt.Errorf("...%v", err) ← %w を使うべき
    - errors.New で元エラーを握りつぶし

[ ] context.Background() の安易な使用:
    - 呼び出しチェーン途中で context.Background() を新規作成（上位の ctx を使うべき）
    - ctx を引数として受け取っているのに使わない

[ ] channel 方向性ミス:
    - 送信専用 chan<- と受信専用 <-chan の使い分けが間違っている
    - goroutine を起動したがチャネルを返していない（goroutine がメインルーティンと断絶）

[ ] nil レシーバ/インターフェース混同:
    - (*T)(nil) を interface に代入して「nil チェック」が通らなくなるパターン
```

### 配線検証（特に重要）

AI はファイル内では正しく実装するが、ファイル横断の結合を忘れがちです。

```bash
# 新規追加された関数・オプション・フィールドが、呼び出し元から実際に渡されているか grep で確認
grep -r "新規追加したフィールド名" --include="*.go"

# フォールバックで常に新機能がスキップされていないか確認
grep -r "?? fallback\|?? default\||| ''" 変更ファイル
```

```
[ ] 新しいパラメータが追加された場合、全呼び出し元から渡されているか
[ ] 新しいインターフェース実装が DI コンテナ/ワイヤリングに登録されているか
[ ] 新しいルート/ハンドラが router に登録されているか
[ ] 新しいマイグレーションが実行順序通りに追加されているか
```

### 既存コードベースとの適合性

```
[ ] 命名規則が既存コードベースのスタイルと一致しているか
[ ] エラーハンドリングスタイルがプロジェクトのパターンと一貫しているか
[ ] ログ出力アプローチがプロジェクトのログ規則を使用しているか
[ ] テストスタイルが既存のテストパターンと一致しているか
[ ] API 呼び出し方式が統一されているか（生成クライアント vs 手書きクライアントの混在なし）
```

### スコープクリープの検出

```
[ ] 要求されていない機能が追加されていないか
[ ] 単一実装のためのインターフェース/抽象化がないか
[ ] 設定可能にする必要のないものを設定可能にしていないか
[ ] 明示的な指示がないのに旧値のマッピング・正規化ロジックを追加していないか
```

### スコープ縮小の検出

```
[ ] タスク要件の取りこぼしがないか
[ ] 実装済みのように見えるが実際には動作しないコードパスがないか
[ ] テストだけあって実装が欠けていないか
```

## 判定手順

1. 変更差分を確認し、AI特有の問題の観点に基づいて問題を検出する
2. 配線検証は grep を実際に実行して確認する（推測でチェックを完了させない）
3. 検出した問題ごとに、Policyのスコープ判定表と判定ルールに基づいてブロッキング/非ブロッキングを分類する
4. ブロッキング問題が1件でもあればREJECTと判定する
