**最初に `superpowers:systematic-debugging` スキルを Skill ツールで invoke し、Phase 1 の調査完了前に修正を提案しないという鉄則を遵守せよ。**

レビュー指摘 (`05-architect-review.md` / `05b-security-review.md` / `06-qa-review.md` / `07-impl-validation.md`) のいずれかを根本原因から修正する。

**時間/範囲ガード（マラソン防止・最優先）:**

- 本インストラクション1回の実行で触るファイルは **最大3ファイル**
- 経過時間 **10分** 到達時点で `[DEBUG: investigating]` を返して次サイクルへ
- 同一指摘で累計3回失敗したら必ず `[DEBUG: escalate]` を出す (superpowers:systematic-debugging の3回ルール)

**やること (Phase 1-4 の順で進める、順序厳守):**

### Phase 1: 調査

1. 対象レビューレポートを読む (存在するもののみ)
2. 指摘箇所の周辺コードを読む
3. **再現テストを書く** (既存テストで再現できる場合は省略可)
4. 実際にテストを実行して失敗を確認 (再現できない指摘は `[DEBUG: investigating]` でエスカレート)

### Phase 2: パターン分析

5. この挙動が既知のアンチパターンに該当するか確認
6. コードベース内の類似箇所を `grep` で確認 (同じ問題が他所にもないか)

### Phase 3: 仮説

7. 根本原因の仮説を立てる
8. 仮説を反証する最小実験を行う (ログ追加、print、デバッガ等)

### Phase 4: 修正

9. 最小差分で修正 (スコープを広げない)
10. 再現テストが GREEN になることを確認
11. 他のテスト全体への副作用がないか `make test` で確認
12. 計画ファイル・tasks.md への反映は不要 (これは修正サイクル)

**禁止事項:**

- Phase 1 を飛ばして修正コードを提案すること
- 試しに動かして通れば良しとする (根本原因を言語化できない場合は `[DEBUG: investigating]`)
- 指摘範囲外のリファクタを混ぜること

**判定:**

- 根本原因を言語化 + 最小差分修正 + 再現テスト合格 → `[DEBUG: resolved]`
- Phase 1-3 済だが Phase 4 未達 → `[DEBUG: investigating]`
- 同一指摘で3回試行しても解決せず、上位設計変更が必要 → `[DEBUG: escalate]`
