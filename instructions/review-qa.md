## レビュー観点

### テストカバレッジと品質

```
[ ] 新しい振る舞いにテストがあるか
[ ] バグ修正にリグレッションテストがあるか
[ ] 正常系があるか
[ ] 入力バリデーション異常系があるか
[ ] 依存エラー異常系があるか
[ ] 境界値テストがあるか（ゼロ値、最大値、空文字列、nil/null）
[ ] 境界値・空コレクション・nil の各ケースが網羅されているか
```

### テスト戦略（単体/統合/E2E）

```
[ ] テストが実装の詳細ではなく振る舞いを検証しているか
[ ] テスト間で状態が漏れていないか（テスト独立性）
[ ] テストの命名が検証内容を明示しているか
```

### Go 並行テスト（Go プロジェクトの場合）

```
[ ] -race フラグ（race detector）でテストが通るか、または通らない根拠があるか
[ ] goroutine リークを検出するテストがあるか（goroutine 数の比較等）
[ ] 並行処理を含む機能のテストが deterministic か
```

### エラーハンドリング（スキルの基準を適用）

```
[ ] エラーを握りつぶしていないか（空の catch / if err != nil { return }）
[ ] エラーが適切にラップされているか（呼び出し元にコンテキストが伝わるか）
[ ] ユーザー向けエラーメッセージが明確か
[ ] 境界変換（HTTP等）にマッピング漏れがないか
[ ] ログへの機密情報出力がないか
```

### リグレッション確認

```
[ ] 既存テストが影響を受けていないか
[ ] 変更により間接的に壊れた機能がないか
```

### 保守性

```
[ ] 関数/ファイルが複雑すぎないか（追いにくい制御フローがないか）
[ ] 命名が明確か
[ ] TODO/FIXME の放置がないか（Issue番号なし）
[ ] 非推奨 API の使用がないか
```

## 設計判断の参照

{report:coder-decisions.md} を確認し、記録された設計判断を把握してください。
- 記録された意図的な判断は FP として指摘しない
- ただし設計判断自体の妥当性も評価し、問題がある場合は指摘する

## 前回指摘の追跡（必須）

- まず「Previous Response」から前回の open findings を抽出する
- 各 finding に `finding_id` を付け、今回の状態を `new / persists / resolved` で判定する
- `persists` と判定する場合は、未解決である根拠（ファイル/行）を必ず示す

## 判定手順

1. 参照スキルファイルを Read で読み込む
2. 前回 open findings を抽出し、`new / persists / resolved` を仮判定する
3. 変更差分を確認し、QA観点（テスト、エラー処理、保守性）に基づいて問題を検出する
   - ナレッジの判定基準テーブル（REJECT条件）と変更内容を照合する
4. 検出した問題ごとに、Policyのスコープ判定表と判定ルールに基づいてブロッキング/非ブロッキングを分類する
5. ブロッキング問題（`new` または `persists`）が1件でもあればREJECTと判定する
