セキュリティの観点から変更をレビューしてください。

## レビュー観点

### OWASP Top 10 チェックリスト

```
[ ] A01: アクセス制御の不備 — 認証なし/認可なしでリソースにアクセスできないか
[ ] A02: 暗号化の失敗 — センシティブデータが平文、弱い暗号、非推奨アルゴリズムで処理されていないか
[ ] A03: インジェクション — SQL/コマンド/LDAP/XPath/NoSQL インジェクション。パラメータ化クエリが使われているか
[ ] A04: 安全でない設計 — 脅威モデリングなし、セキュアデザインパターン未適用
[ ] A05: セキュリティの設定ミス — デフォルト認証情報、不必要に有効な機能、エラーメッセージ過多
[ ] A06: 脆弱な/古いコンポーネント — 既知CVEを持つライブラリの使用
[ ] A07: 認証・セッション管理の不備 — ブルートフォース無対策、セッション固定、弱いトークン
[ ] A08: ソフトウェアとデータの整合性の失敗 — 検証なしのデシリアライズ、CI/CDパイプライン汚染
[ ] A09: セキュリティログとモニタリングの失敗 — ログ不足、機密情報のログ出力
[ ] A10: サーバーサイドリクエストフォージェリ (SSRF) — ユーザー入力のURLへの直接アクセス
```

### Go 固有セキュリティ

```
[ ] unsafe パッケージの使用がないか（正当な理由なし）
[ ] ユーザー入力が直接 exec.Command / os/exec に渡されていないか（コマンドインジェクション）
[ ] ファイルパスにユーザー入力が含まれる場合、path traversal 対策があるか
[ ] struct tag（json/yaml）に機密フィールドが誤って含まれていないか
[ ] http.ListenAndServe に timeout が設定されているか
[ ] TLS 設定で MinVersion が適切か（TLS 1.2 以上）
[ ] 乱数生成に crypto/rand を使用しているか（math/rand は使用しない）
[ ] SQL クエリに fmt.Sprintf / string 連結を使っていないか
```

### AI 固有セキュリティアンチパターン

```
[ ] 幻覚した暗号パターン — 標準ライブラリに存在しない暗号関数の呼び出し
[ ] 過度に寛容な CORS — AllowOrigin: "*" + AllowCredentials: true の組み合わせ
[ ] insecure defaults — 開発用設定がプロダクションコードに混入
[ ] 機密情報のハードコード — API キー、パスワード、トークンが直書きされていないか
[ ] エラーメッセージへの内部情報漏洩 — スタックトレース、DBスキーマ、実装詳細を含むエラー
```

### データ保護

```
[ ] PII/機密データが不必要にログに記録されていないか
[ ] センシティブデータが適切にマスク/ハッシュ化されているか
[ ] 認可チェックが実装ロジック側にあるか（フロント側だけでない）
```

## 前回指摘の追跡（必須）

- まず「Previous Response」から前回の open findings を抽出する
- 各 finding に `finding_id` を付け、今回の状態を `new / persists / resolved` で判定する
- `persists` と判定する場合は、未解決である根拠（ファイル/行）を必ず示す

## 判定手順

1. 変更差分を確認し、上記セキュリティ観点に基づいて問題を検出する
   - ナレッジの判定基準テーブル（REJECT条件）と変更内容を照合する
2. 検出した問題ごとに、Policyのスコープ判定表と判定ルールに基づいてブロッキング/非ブロッキングを分類する
3. ブロッキング問題（`new` または `persists`）が1件でもあればREJECTと判定する

**セキュリティ問題は自動的に REJECT:** インジェクション、認証不備、機密情報ハードコード、SSRF は発見次第必ずREJECT（Warningにしない）。
