# PRバッチサイズ規約

SDDの実装を「小さめのPR単位」で完了させ、セッションのコンテキスト消費を抑えるための規約。

## 目的

- 1回の `sdd-impl` / `sdd` 実行で扱う実装範囲を絞る
- レビューしやすいPRサイズを維持する
- 残タスクがあっても「今回バッチ完了」でセッションを終了可能にする

## 原則

### 1. 1バッチ = 1目的（原則）

- 1つのPRバッチでは、1つの価値または1つの縦スライスに集中する
- 例:
  - preflightの導入
  - 共通fixtureの最小版
  - 最小E2E 1ケース（local/stg共通）

### 2. 大規模変更は分割する

以下は別PRバッチに分割する:
- 広範囲の既存spec移行
- 実装基盤 + CI統合の同時実施
- リファクタリングと機能追加の同時実施

### 3. 最小縦スライスを優先する

- 全部の一般化より先に、1ケース通る最小構成を完成させる
- 「使える最小版」を早く出す

## 推奨サイズ（目安）

厳密な制限ではなく、レビュー容易性のための目安。

- 変更ファイル数: 10〜15以内（生成物・lockfile除く）
- 実質変更行数: 300〜600行以内（概算）
- 新規概念: 1〜2個まで

上限を超える場合は、次バッチへ送ることを優先する。

## tasks.md での表現ルール

- `tasks.md` には `PRバッチ計画` セクションを含める
- 各バッチは `PR-1`, `PR-2` ... の識別子を持つ
- 各バッチには最低限以下を定義する:
  - 目的
  - 対象タスクID
  - 完了条件

## 実装計画（plan）での適用ルール

- `PRバッチ計画` がある場合、planはまず未完了の最初のPRバッチを候補とする
- 候補バッチが大きすぎる場合は、同一PRバッチ内の部分集合を「今回のバッチ」として選んでよい
- ただし別PRバッチのタスクを混在させない

## セッション完了ルール

- 全タスク未完了でも、今回PRバッチの完了条件を満たしたら「PRバッチ完了」で終了してよい
- 残タスクは次回セッションで継続する
