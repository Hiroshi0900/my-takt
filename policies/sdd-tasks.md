# SDDタスク生成規約

仕様駆動開発におけるタスク分解のルール。

## 原則

| 原則 | 基準 |
|------|------|
| 自然言語記述 | 機能・振る舞い・成果で記述。ファイルパス・関数名・型名は禁止 |
| 要件マッピング | 各タスク末尾に `_Requirements: X.X, Y.Y_` を付与 |
| コードのみ | 実装・テスト・技術セットアップのみ。デプロイ・ドキュメント・ユーザーテストは含まない |
| 段階的構築 | 前のタスクの成果に基づき次を積み上げる |
| PRバッチ化 | 実装タスク一覧とは別に `PRバッチ計画` を定義し、小さめPR単位で完了できるようにする |

## タスク階層

最大2レベルまで。

```markdown
- [ ] 1. メジャータスク
- [ ] 1.1 サブタスク
  - 詳細項目
  - _Requirements: 1.1, 1.2_
- [ ] 1.2 サブタスク
  - 詳細項目
  - _Requirements: 1.3_
```

3レベル以上のネストは禁止。

## 並列分析

タスクが並列実行可能な場合、タスク番号の直後に `(P)` を付与する。

```markdown
- [ ] 2.1 (P) バックグラウンドワーカーの構築
```

### 並列条件（全て満たすこと）

- 未完了タスクへのデータ依存がない
- ファイル・共有リソースの競合がない
- 前提レビュー・承認が不要
- 環境・セットアップが完了済み

## サイズ基準

論理的なまとまりでグルーピングする。人為的な分割数に合わせない。

## PRバッチ計画（必須）

`tasks.md` には、実装タスク一覧の前に `PRバッチ計画` セクションを含める。

### バッチ記法（例）

```markdown
## PRバッチ計画

### PR-1: 最小縦スライス
- 目的: local/stg 共通で1ケース実行できる土台を作る
- 対象タスク: 1.1, 1.2, 2.1, 2.2, 2.3
- 完了条件: 共通ケースIDで local/stg 実行経路が成立する

### PR-2: 共通fixtureの拡張
- 目的: 監視・skip理由・成果物出力を共通化する
- 対象タスク: 3.1, 3.2, 4.1, 4.2, 4.3
- 完了条件: pageerror/console監視と skip理由出力が機能する
```

### バッチ分割ルール

- 原則として `1バッチ = 1目的`
- 別バッチのタスクを混在させない
- 先に「最小縦スライス」を置く
- CI統合や大規模移行は将来バッチへ分離する

## 禁止事項

- ファイルパス・関数名・クラス名・API契約の記述
- 3レベル以上のネスト
- デプロイ・ドキュメント・ユーザーテストのタスク
- 要件IDの省略
- `PRバッチ計画` の欠落
